- **Bugfix: aggressive HPA scaling no longer OOM-kills scale pods**: the generated HorizontalPodAutoscaler previously emitted only a CPU metric (a configured `memory` trigger was silently rewritten to CPU) and carried no `behavior` block, so the Kubernetes default of doubling the fleet every 15s could burst pods faster than nodes absorb their memory. Every cpu/memory trigger now maps to its own metric, a default memory trigger (80% of the container memory request, `memory_utilization_target`) scales pods out before they hit their memory limit, and both engines (HPA, and KEDA via `advanced.horizontalPodAutoscalerConfig`) ship behavior guardrails: scale-up stabilization plus a pods-per-minute rate limit (`autoscaler_scale_up_stabilization` / `autoscaler_scale_up_max_pods`), and scale-down stabilization from `autoscaler_cooldown` with a 50%-per-minute cap. Per-service `hpa.memory_target` overrides are supported, `jac scale doctor` validates them, and the microservice e2e asserts the guardrails on every managed HPA.
